# 面试与简历包装

## 1. 一句话介绍

VideoCraft Agent 是一个 AI 营销短视频生成助手。用户输入产品信息并上传素材后，系统通过 Dify Workflow 生成短视频分镜脚本，再由 FastAPI 编排本地素材匹配和 FFmpeg 合成，最终输出带字幕的 9:16 竖屏营销视频。

## 2. 简历写法

可以写成：

```text
独立开发 AI 营销短视频生成工具，采用 Dify Workflow + FastAPI + FFmpeg 架构；基于 Dify 实现短视频脚本、分镜、字幕和素材关键词生成，基于 FastAPI 实现素材上传、任务状态管理、Dify API 调用和结果校验，基于 FFmpeg 实现素材裁剪、竖屏适配、片段拼接、字幕叠加、背景音乐合成和视频格式转换；设计“文本需求 / 上传素材 -> AI 分镜脚本 -> 后端编排 -> 视频合成 -> 预览下载”的自动化流程，适用于营销短视频和产品介绍视频的快速草稿生产。
```

如果需要更短：

```text
开发 AI 营销短视频生成工具，使用 Dify Workflow 生成结构化分镜脚本，结合 FastAPI 文件上传与任务编排，并通过 FFmpeg 自动合成 9:16 竖屏视频，支持预览、下载和格式转换。
```

## 3. 面试 1 分钟讲法

可以这样讲：

```text
我做了一个 AI 营销短视频生成工具。它的目标不是训练底层视频模型，而是先解决业务里更常见的短视频制作流程自动化问题。用户输入产品名称、目标用户、卖点和平台，并上传素材后，FastAPI 后端会调用 Dify Workflow 生成 3 段式分镜脚本，再根据脚本里的关键词匹配上传素材或本地素材库，最后用 FFmpeg 自动裁剪、拼接、加字幕和背景音乐，输出一个 9:16 的 mp4 视频。

这个项目里我把 Dify 用在 AI 脚本和关键词生成，把 FastAPI 用在任务编排、文件处理和状态管理，把 FFmpeg 用在确定性的视频处理。这样既能展示 AI 工作流能力，也能体现后端工程和视频处理能力。
```

## 4. 和 JD 的对应关系

JD: AI 视频生成工具

项目对应：

- 实现从文本需求和上传素材到短视频成片的工具。
- 虽然 MVP 不训练模型，但完成了视频生产自动化闭环。

JD: 文本 -> 视频

项目对应：

- 用户输入文本业务需求。
- Dify 输出结构化分镜脚本。
- FastAPI 和 FFmpeg 输出短视频文件。

JD: 素材库和剪辑工具

项目对应：

- 支持上传素材和本地素材库兜底。
- FFmpeg 自动裁剪、拼接、加字幕、转格式。

JD: AI Agent / Workflow

项目对应：

- Dify Workflow 负责 AI 脚本生成。
- FastAPI 将任务拆分为上传、脚本、素材、合成、导出多个节点。
- AI 负责创意决策，工具负责确定性执行。

JD: Python / FastAPI / FFmpeg

项目对应：

- 使用 Python 和 FastAPI 实现后端服务。
- 使用 FFmpeg 实现视频处理。
- 使用 Pydantic 校验 AI 返回结构。

JD: 业务理解

项目对应：

- 聚焦营销短视频和产品介绍视频。

## 5. 为什么不直接做复杂文生视频模型

推荐回答：

```text
我没有一开始就把重点放在训练或深度调优文生视频模型，因为这个岗位描述更强调把现成 AI 能力和业务流程结合起来。我先做了一个业务可用的自动化视频生产工具，把 Dify 的 AI 脚本生成、FastAPI 的任务编排、素材匹配、字幕、合成和格式转换跑通。这样能更快交付一个可演示工具，也方便后续把开源文生视频模型接成“缺少素材时自动生成片段”的增强节点。
```

## 6. 可以强调的技术点

- Dify Workflow 输出 JSON，而不是纯文本，方便系统解析。
- Pydantic 校验脚本结构，提高稳定性。
- FastAPI 统一管理上传文件、任务状态和 Dify API Key。
- 用户上传素材优先，本地素材库兜底。
- 用 FFmpeg 处理视频比例、裁剪、拼接、字幕、音乐和格式转换。
- 设计失败兜底，Dify 失败时仍可用默认脚本继续演示。
- 前端只负责交互，不直接调用 Dify，避免 API Key 泄露。

## 10. 面试官可能追问

### Q1: 这个项目哪里体现 Agent 或 Workflow？

回答要点：

- 用户只给业务目标和素材。
- Dify Workflow 把需求转成分镜、字幕和关键词。
- FastAPI 把任务拆解为上传、脚本、素材、合成、导出多个步骤。
- AI 负责生成和判断。
- FFmpeg 等工具负责确定性执行。
- 每个节点都有输入输出和失败处理。

### Q2: 为什么素材匹配先用关键词？

回答要点：

- MVP 优先保证闭环。
- Dify 已经能生成 `visual_keywords`。
- 关键词足够演示业务流程。
- 后续可以升级成 embedding 向量检索。

### Q3: Dify 返回不稳定怎么办？

回答要点：

- Dify Prompt 要求只输出 JSON。
- Workflow 中使用 Code 节点清洗字段。
- 后端用 Pydantic 校验。
- 解析失败时重试。
- 仍失败则使用模板脚本兜底。

### Q4: 为什么 FastAPI 调 Dify，而不是 Dify 调 FastAPI？

回答要点：

- MVP 中 FastAPI 能访问本地上传文件和 FFmpeg。
- Dify API Key 放在后端更安全。
- 任务状态由 FastAPI 统一管理。
- 后期如果服务部署到公网，可以用 Dify HTTP Request 节点反向调用 FastAPI。

### Q5: FFmpeg 出错怎么排查？

回答要点：

- 记录完整命令。
- 捕获 stderr。
- 检查素材路径、编码、分辨率、字体路径。
- 必要时把处理拆成多个中间文件。

### Q6: 如果要批量生成 100 条视频怎么办？

回答要点：

- 需求输入改为 CSV / Excel。
- 每条任务保存状态。
- 视频生成放入队列。
- 素材和中间结果缓存。
- 页面展示任务列表和失败重试。
